# source-scripts

This case study was originally produced from a private CRM export. That production code (CRM API sync, field flattening, anonymisation) is not redistributable, and the anonymised source CSVs are not included here.

So the portfolio version is fully self-contained and synthetic. It takes no private input:

- `../python/build_dimensional_model.py` — a seeded synthetic generator. It designs a channel propensity model (channel is the primary close signal, firmographic ICP a secondary tilt, anti-ICP industries pinned near zero), pins Cold Calling to its designed win rate with one bisection-calibrated global multiplier so the channel ordering reproduces on every run, and writes the fully fictionalised star schema to `../data/`. It also runs a **per-channel post-won survival model**: each won deal draws a lifetime from a geometric process whose monthly survival solves to the channel's designed 12-month retention anchor (`s = m12 ** (1/12)`), producing `is_churned` / `retained_months` / `churn_date_key` / `churned_mrr`. The churn draws use dedicated independent RNG streams (`SEED+900`), so they are appended *after* the existing draw sequence and the pre-churn columns (`deal_key … is_lost`) remain byte-identical — the proven win/volume/MRR numbers did not move when retention was added (verified: first-16-column diff empty).
- `../python/verify_numbers.py` — recomputes every quantitative claim in the README, deck, and dashboard spec from the generated CSVs. This is the canonical-numbers source of truth.
- `../python/analysis.py` — reproduces the five headline charts.
- `../sql/schema.sql` + `../sql/queries.sql` — DDL and 10 analytical queries, validated against the generated CSVs in DuckDB (zero foreign-key orphans on all joins including the new `churn_date_key`; Q10 scorecard band rollup cross-checked against `verify_numbers.py` and matches exactly).
- `../powerbi/` — a working PBIP/TMDL Power BI project plus its spec, generated by `_gen_tmdl.py` / `_gen_report.py` and validated headless with pbi-cli.

Every figure in this case study is synthetic and deliberately round. It does not correspond to any real client's results. The blended win rate is intentionally high so the dialer-versus-warm contrast is legible; retention comes from a designed survival model, not a fitted one, and small-n channels are flagged directional everywhere. The purpose is to demonstrate the method (channel-level dimensional modelling, a post-sale retention axis, return per scarce resource, an allocation scorecard that reproduces its own bands, BI delivery), not to report a real engagement.

No real company names, person names, emails, or phone numbers appear anywhere in `../data/` or any deliverable. Company names are Faker-generated; channels and reps are fictional labels.
